# 🧠 usecase/ (สมองของแอปพลิเคชัน)

## 📌 หน้าที่
ที่นี่คือที่ตั้งของ **Business Logic (กฎเกณฑ์ทางธุรกิจ) ทั้งหมด** 
ถ้าบริษัทสั่งมาว่า "รหัสผ่านต้องยาว 8 อักขระขึ้นไป" หรือ "ค้นหาได้เฉพาะบทความของตัวเองเท่านั้น" โค้ดที่ว่าต้องมาอยู่ที่นี่!

Usecase เป็นตัวประสานงาน (Orchestrator) ระหว่างฝั่ง `Delivery` กับฝั่ง `Repository`
มันรับคำสั่งมาจาก `Delivery` -> ตรวจสอบ Business Rules -> ใช้ `Repository` ไปดึง/บันทึกข้อมูล -> ปั้นข้อมูลส่งตลับไป

## ❌ ข้อห้าม
- **ห้ามยุ่งกับโลกภายนอกเด็ดขาด** : ห้ามเรียกใช้ `http.Request`, ห้ามรับ JSON ตรงๆ รูปร่างหน้าตาของหน้าเว็บจะเป็นยังไง Usecase ไม่สน มันรับแค่ Data struct เปล่าๆ เพียวๆ
- **ห้ามคุยกับ DB โดยตรง** : เช่น `db.Query(...)` ห้ามอยู่ที่นี่! ถ้าจะคุยต้องคุยผ่าน *สัญญา (Interface)* ของ `Repository` เท่านั้น (ทำให้เราเทส (Unit Test) โค้ดง่ายขึ้นแค่นำ Mock มาจำลองฐานข้อมูล)

## 💡 ตัวอย่าง
ใน `auth_usecase.go` เมื่อมีคนเรียก `Login(username, password)`:
1. เรียก `UserRepository` ให้ไปหา `username` ว่ามีจริงไหม (แต่เราไม่สนหรอกว่า Repo ไปหาด้วย SQL อะไร)
2. ถ้าหาเจอก็เอา `password` ที่เขาป้อนมา มาเทียบ (Hashing) กับ Password ที่คืนมาจาก Repo
3. ถ้าตรงกัน ก็อนุญาตให้สร้าง JWT Token และส่งคืน
